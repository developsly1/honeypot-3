Static Site Deployment Notes

These notes collect the practical details we revisit each time we ship a static site to a new host. They are intended as a quick reference rather than an exhaustive guide, and they assume a build step that produces plain HTML, CSS, and JavaScript into an output directory.

Build output and paths

Most static generators emit to a folder such as dist, build, or public. The single most common deployment mistake is a mismatch between where the generator writes files and where the host expects to serve them. When a site is served from a subpath rather than a domain root, asset references must be relative or prefixed with the base path, otherwise stylesheets and scripts resolve to the wrong location and the page renders unstyled.

Cache headers

Hashed asset filenames can be cached aggressively because their names change whenever their contents change. HTML documents should use a short cache lifetime or revalidation so that updated markup is picked up promptly. Mixing the two — long-lived HTML with hashed assets — is a frequent cause of users seeing stale pages after a deploy.

Status codes and redirects

A correct deployment returns a 200 for existing pages and a real 404 for missing ones. Soft 404s, where a missing page returns 200 with a friendly message, confuse crawlers and analytics. Permanent moves should use a 301, temporary ones a 302. Redirect chains longer than one hop add latency and should be collapsed where possible.

Verification

After any deploy, confirm the entry page and one deep link both load, check that assets return 200, and verify the canonical link points where you expect. A two minute check catches most regressions before anyone else sees them.

Reference notes only. Host behavior varies; consult your provider's documentation for specifics.
